=== Slide 11 ===
When you run an INSERT/UPDATE/DELETE, Postgres modifies pages in
shared buffers
(memory), not the data files on disk directly.
Every change is also appended to a sequential log file called the
Write-Ahead Log (WAL)
.
Postgres only considers a
COMMIT
successful once the relevant WAL records are safely flushed to disk, so it can always replay them after a crash.
Periodically, Postgres performs a
checkpoint
: it writes all dirty (modified) pages from memory back to the main data files on disk.
The WAL plus the latest checkpoint let Postgres reconstruct the exact state of the database if it restarts.
​Updates and deletes don’t overwrite rows in place; they create new row versions and mark old ones as dead, to support concurrent transactions seeing consistent snapshots (
MVCC
).
A background
vacuum
process later cleans up those dead rows.
Persisting data modification in PostgreSQL

=== Slide 16 ===
Autoscaling
Define minimum and maximum compute size
Fully automated dynamic resource allocation
No downtime during scaling operations
Vertical bidirectional scaling in increments of 1 CU (2GB of RAM)
Can scale in both directions in ~500ms
Scaling decision based on CPU load average, memory usage, number of unique page accessed in the cache during the last hour

=== Slide 17 ===
A branch is a lightweight, copy-on-write clone of your database created from current or historical data and running in an isolated Postgres instance
It inherits both the schema and data from its parent, but shares the same underlying storage
A new branch references its parent’s data until changes are made. Then, only the diffs are written
That means branches are fast to create regardless the dataset size, cost-efficient to run, and safe to discard.
Branches
Main dataset

=== Slide 18 ===
Snapshots capture the state of your branch at a point in time, which can be specified either using a timestamp or a LSN
They can be created manually, or automatically on schedule
Schedule applies on a branch level and only applies to root branches
Snapshots can be preserved for longer than the history retention period defined for the project
Restoring from snapshot is nearly instantaneous (~1sec)
Snapshots
Main dataset

Allows restoring a database to any point in time within the database project's restore window
The history retention period can be configured for up to 30 days
Extending retention increases storage usage but provides greater flexibility
A restore operation typically completes in a few seconds
During a restore operation, a backup of your current database is automatically created

AUTO (default)

FULL
- Always perform full refreshes. Databricks never performs incremental refresh, even when the query is incrementalizable.

FAQ: Why is catalog federation read-only?
Unreliable pattern for enterprise production workloads
UC cannot guarantee reliability or write correctness in foreign catalogs
E.g. commit issues in 3p catalog can cause failures with long-running jobs, or changes in the foreign catalog can cause upstream issues in UC
Bi-directional read federation is the standard – other vendors including AWS Glue, MSFT OneLake, and BigQuery have embraced
Bottom line
Bi-directional read federation is the industry standard for interoperability.

=== Slide 24 ===
Databricks Web UI and REST API access is authenticated via Databricks Identity Management.
Clients can connect to Lakebase directly using the PostgreSQL Wire Protocol, which supports Databricks Identity and local authentication.
Administrative operations are authorized and routed to the Lakebase Management Service (LBM). These requests use internal tokens and are transmitted over TLS 1.2 or higher.
SQL commands issued via notebooks, the Lakehouse SQL Editor are authorized through Unity Catalog or native Postgres and forwarded to the Lakebase server over TLS 1.2+.
System-initiated actions, such as Reverse ETL jobs and model serving requests, are authenticated using temporary OAuth tokens. All communications occur over TLS 1.2+.
Lakebase Compute nodes interact with the Lakebase Storage Service to access and serve OLTP data.
The Lakebase Storage Service manages data persistence, including long-term storage and encryption.
